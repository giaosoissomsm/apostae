<objective>
Extend Phase 1's `notificationService.register()` with one additive listener for `wager.cashed_out`, reusing the existing chokepoint per ROADMAP.md's Phase 2 dependency note ("reuses the notification chokepoint to emit a cashout-confirmed event") — and critically, using a cashout-row-scoped `relatedId` (not the wager's id) so repeated cashouts on the same wager each produce their own notification instead of silently colliding with the existing idempotency UNIQUE constraint.

Purpose: RESEARCH.md Pitfall 3 identifies a real, specific bug risk: if this listener copies the other 7 catalog events' `relatedId: evt.wagerId` shape verbatim, the second cashout on the same wager would silently produce no notification (swallowed by the existing 23505-catch-as-no-op). This plan exists specifically to get that one field right and prove it with a test.
Output: One new `domainEvents.on('wager.cashed_out', ...)` listener block; a test proving two distinct cashouts on the same wager produce two distinct notification rows.
</objective>
