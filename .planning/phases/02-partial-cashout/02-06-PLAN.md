<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request body → controller | Fully attacker-controlled; only amount/idempotency_key are meaningful, everything else must be discarded |
| JWT session → req.user.id | Trusted (already authenticated by requireAuth); the only valid source of userId |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-06-01 | Elevation of Privilege | mass assignment (body-supplied userId/wagerId) | high | mitigate | Controller destructures only `{ amount, idempotency_key }`; wagerId comes from `req.params.id`, userId comes exclusively from `req.user.id` — verified by controller test case (2) |
| T-02-06-02 | Tampering | parameter tampering (body-supplied payout/value field) | high | mitigate | Controller never reads any field other than amount/idempotency_key; the service always computes the payout server-side (Plan 02-04) — verified by controller test case (3) |
| T-02-06-03 | Spoofing | unauthenticated cashout attempt | high | mitigate | `requireAuth` middleware (existing, unchanged) gates the route — no new auth bypass surface introduced |
</threat_model>
