  <behavior>
    - Given a pending wager in an open market with cashed_out_amount = 0 and amount 100 at CANCEL_FEE_PERCENT 5: net refund = 95, fee = 5, wager status becomes 'refunded', one wallet_transactions 'refund' row for 95 (CANCEL-01, CANCEL-02, CANCEL-05, CANCEL-04)
    - Given a wager with cashed_out_amount > 0: throws ConflictError before any refund math, wallet unchanged, status unchanged (CANCEL-06)
    - Given a closed market or a market past closes_at: throws ConflictError (CANCEL-06)
    - Given a non-pending wager (already resolved/refunded): throws ConflictError (CANCEL-06)
    - Given a wagerId owned by a different user: throws NotFoundError (404, not 403) because the ownership-scoped locked SELECT returns null (CANCEL-07 access control)
    - The market row is locked (marketRepository.findByIdForUpdate) before the wager row (wagerRepository.findByIdForUpdate) before the wallet row (CANCEL-07 deadlock fix)
  </behavior>
  <action>
Replace the body of `cancelWager(wagerId, userId)` (currently the version that locks wager-before-market via raw `SELECT * FROM wagers ... FOR UPDATE`, checks user_id afterward, and never blocks on cashout) with a structure that mirrors `cashoutWager`'s proven lock sequence in the same file. Follow the recommended shape in 04-PATTERNS.md / 04-RESEARCH.md, verifying field and error-message wording against the live file.

Required steps inside the `transaction(async (client) => { ... })` body, in this order:
1. Unlocked peek to get market_id (market_id is immutable on a wager, so this is not a TOCTOU risk): `SELECT market_id FROM wagers WHERE id = $1`. If no row, throw NotFoundError.
2. Lock the market FIRST via `marketRepository.findByIdForUpdate(peek market_id, client)` (fixes D-04a lock order). If missing or status !== 'open', throw ConflictError (market closed). If closes_at is set and passed, throw ConflictError (deadline passed). Keep the existing Portuguese messages.
3. Lock the wager SECOND via `wagerRepository.findByIdForUpdate(wagerId, market.id, userId, client)` — ownership baked into the WHERE clause (fixes D-04b IDOR). If it returns null, throw NotFoundError (404, avoids leaking existence to a non-owner). If wager.status !== 'pending', throw ConflictError.
4. CANCEL-06 hard block (D-04c): if `Number(wager.cashed_out_amount) > 0`, throw ConflictError with a message stating the wager already had a cashout — this MUST run before any fee/refund computation.
5. CANCEL-03 remaining-stake formula: `const remainingStake = Number(wager.amount) - Number(wager.cashed_out_amount)`. Given step 4, this always equals wager.amount in practice; keep it as the defensive formula consistent with resolveMarket/deleteMarket. Do not implement partial-cancel-after-cashout (per D-01 it is unreachable).
6. Compute the split via `money.applyFeePercent(remainingStake, env.CANCEL_FEE_PERCENT)` -> `{ fee, net }`. `net` (95%) is refunded; `fee` (5%) is simply not refunded (never credited elsewhere).
7. `wagerRepository.updateStatus(wagerId, 'refunded', client)` — reuse the existing 'refunded' status value; do NOT add a migration or a new 'cancelled' enum value (dashboard.js already labels 'refunded' as "Cancelada" — CANCEL-05).
8. Lock the wallet THIRD via `walletRepository.findByUserIdForUpdate(userId, client)`; capture balanceBefore; `adjustBalance(wallet.id, net, client)`; then `recordTransaction` with type 'refund', amount `net`, populated balanceBefore/balanceAfter, relatedEntity 'wager', relatedId wager.id, and a description documenting gross/fee/net (e.g. mentions the fee percent and net refund over the remaining stake) for audit readability (CANCEL-04).
9. Return `{ marketId: market.id, question: market.question, grossAmount: remainingStake, feeAmount: fee, netAmount: net }`.

After the transaction resolves (not inside it), keep `domainEvents.emit('wager.cancelled', {...})`. Preserve the existing `amount` field semantics by setting `amount: result.netAmount` (notificationService's existing listener reads evt.amount); also include grossAmount and feeAmount. Keep a logger.info line. Return `{ ok: true, refunded: result.netAmount, fee: result.feeAmount }`.

Because ownership is now enforced by the locked SELECT returning null, the rewrite no longer throws AuthorizationError. Remove `AuthorizationError` from the destructured require on line 11 of this file — it is used nowhere else in wagerService.js (confirmed: only the old cancelWager referenced it). Do not touch placeWager or cashoutWager. Do not add any request-body field to the flow — cancelWager takes no client-supplied amount/fee/id beyond the route param wagerId and the JWT userId.
  </action>
