| client HTTP → notification routes | Authenticated request may attempt to access another user's notification by id |
| client query params → repository | `limit`/`page`/`unread_only` are untrusted input |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Information Disclosure / Tampering | PATCH /:id/read, GET / | high | mitigate | requireAuth on every route; identity from req.user.id only; service+repo scope every query by userId; 404 (not 403) on foreign id (Tasks 1-2), verified by ownership test |
| T-01-04 | Denial of Service (resource exhaustion) | GET / limit param | medium | mitigate | Server-side limit cap (Plan 02 repo) re-verified by pagination test asserting >100 is clamped |
| T-01-05 | Spoofing | identity source | high | mitigate | User id never accepted from params/body (grep gate returns 0); only the JWT-derived req.user.id is trusted |
</threat_model>
