# Phase 3 — UI Design Contract

> Visual and interaction contract for the New Market Types phase. Generated by gsd-ui-researcher,
> verified by gsd-ui-checker. No `03-CONTEXT.md` exists (discuss-phase was skipped this
> milestone) — every decision below is sourced from `ROADMAP.md`, `REQUIREMENTS.md`,
> `STATE.md`'s two locked Phase 3 decisions, and `03-RESEARCH.md`. Decisions this document makes
> unilaterally (not already locked upstream) are flagged **[UI-SPEC discretion]** with rationale.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — plain HTML/CSS/vanilla JS (Express-served static `public/`), not a React/Next.js/Vite project. shadcn initialization gate does not apply. |
| Preset | not applicable |
| Component library | none (hand-rolled CSS component classes in `public/css/style.css`) |
| Icon library | none — emoji glyphs used sparingly (💰, ⏱️) as existing convention; no icon font/SVG set. Phase 3 introduces zero new icon usage. |
| Font | Space Grotesk (display/headings), Inter (body), IBM Plex Mono (numeric/mono) — loaded via Google Fonts `<link>` in each HTML page, already established, unchanged this phase |

**Existing design system is fully formed** (`public/css/style.css`, 412 lines, dark theme with CSS custom properties). Phase 3 adds **zero new design tokens** — every color, spacing exception, and typographic role below is a reuse of what's already shipped. New elements introduced this phase (market-type selector, dynamic option-row builder, generalized N-option ticket buttons) must be built from existing classes (`.field`, `.inline-form`, `.btn-ghost`, `.btn-primary`, `.odd-btn`, `.tag`, `.panel`) plus a small number of explicitly listed new modifier classes (see Component Inventory).

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing — new `.options-list` row gap, `.option-row` internal gap |
| md | 16px | Default element spacing — `.grid-2` gap (existing, reused for new fieldsets) |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Exceptions (pre-existing, documented, not to be extended):** the current codebase does not strictly follow an 8pt scale — established component classes use fine-grained even values outside multiples of 4 (`.ticket-top` padding `20px 20px 16px`, `.panel` padding `22px`, `.odd-btn` padding `10px 8px`, `.credits-pill` padding `6px 14px`, gaps of `6px`/`10px`/`14px`/`18px`/`22px`/`26px` scattered across `.topbar`, `.field`, `.auth-card`). **Phase 3 rule: reuse these existing classes verbatim wherever the same visual role is needed — do not hand-roll new spacing values for anything that already has an established class.** The only genuinely new spacing decision this phase introduces is the `.options-list`/`.option-row` gap, which uses `sm` (8px), matching `.inline-form`'s existing `gap: 8px`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#14161c` (`--bg`) | Page background |
| Secondary (30%) | `#1d2028` (`--surface`) / `#262a35` (`--surface-2`) | Panels, tickets, cards, form inputs, table rows |
| Accent (10%) | `#c6ff3d` (`--sim`) | See "Accent reserved for" below |
| Destructive | `#ff5c7a` (`--nao`) | Destructive/negative actions and states only |

**Accent (`--sim`) reserved for:**
- `.btn-primary` backgrounds ("Criar mercado", "Apostar" — unchanged, reused verbatim for all 3 market types)
- The **selected** state of any option button, binary or generalized (`.odd-btn.selected` border + tinted background) — this is the single unifying accent treatment for "the option the user has currently chosen," regardless of market type
- Pre-existing, unchanged: brand-mark dot, `.ticket-status.open`, binary `.odd-btn.sim` value color

**Destructive (`--nao`) reserved for:**
- Remove-option-row buttons in the multiple-choice builder (small "×" button, styled exactly like the existing "Deletar" market button: `color: var(--nao); border-color: var(--nao-dim);`)
- Pre-existing, unchanged: binary `.odd-btn.nao` value color, error toasts/messages, "Deletar" buttons

**[UI-SPEC discretion] New generalized option buttons (Over/Under and multiple-choice) do NOT get per-option accent colors.** Binary's Sim=lime/Não=red duality is a *semantic* affirmative/negative color pairing that does not generalize to N arbitrary options (e.g. "Time A" vs "Time B" — neither is inherently positive or negative), and Over/Under is not a value judgment either ("Over" is not "good," "Under" is not "bad"). Assigning per-option hues would also blow the 10%-accent budget once option count exceeds 2. **Rule:** every non-binary option button uses a neutral value color (`var(--text)`) at rest; the *only* color signal is the shared `--sim` "selected" treatment applied uniformly regardless of which option is chosen. This preserves MARKET-03's pixel-identical requirement (binary's `.odd-btn.sim`/`.odd-btn.nao` classes and colors are completely untouched) while giving new types a single, consistent, budget-respecting selection affordance.

`--gold` (`#f5c84c`) is a pre-existing third informational accent (credits amount, focus ring, `.tag.admin`, closed-status badge) — outside the 10% interactive-accent budget, used only for informational/status highlighting. Phase 3 reuses it unmodified for the existing `.ticket-status.closed` styling; no new `--gold` usage is introduced.

## Admin Panel — Market Creation Form Contract

**Field/label copy (PT-BR, matching existing informal tone):**

| Field | Label | Notes |
|-------|-------|-------|
| `#mMarketType` | "Tipo de mercado" | Options: "Binário (Sim/Não)" (default, selected), "Over/Under (limite numérico)", "Múltipla escolha" |
| `#mQuestion`, `#mDescription` | "Pergunta", "Descrição (opcional)" | Unchanged, shared across all 3 types |
| Binary fieldset | "Odd Sim" / "Odd Não" | **Unchanged** — `#mOddsYes`/`#mOddsNo`, identical to today |
| Over/Under fieldset | "Limite (ex: 2.5)" / "Odd Over" / "Odd Under" | `#mThreshold` (number, step 0.5, min 0.5), `#mOddsOver`, `#mOddsUnder` (number, step 0.01, min 1.01, matching existing odds inputs) |
| Multiple-choice fieldset | "Opções" (section label) | Dynamic rows in `.options-list`; each row = text input `placeholder="Ex: Time A"` + odds number input `placeholder="Odd"` (step 0.01, min 1.01) + remove button. No per-row `<label>` element, matching the existing label-less `.inline-form` convention already used in "Novo usuário." |

**[UI-SPEC decision, resolving RESEARCH.md Open Question 2]:** Over/Under gets **3 dedicated inputs** — one threshold + two odds fields (Odds Over, Odds Under) — not a bare 2-input form. Rationale: STATE.md's locked decision says the threshold input "auto-generates the 'Over X'/'Under X' market_options pair" — it auto-generates the **labels**, not the **odds**. Every market type in this system is fixed-odds (REQUIREMENTS.md Out of Scope explicitly rules out pari-mutuel/pool payouts), so `market_options.odds` (`NOT NULL` per migration 005) must be admin-supplied at creation time. RESEARCH.md's own framing of Over/Under as "structurally identical to binary's odds_yes/odds_no pair, just with a threshold and different labels" directly supports mirroring binary's 2-odds-field pattern plus the one new threshold field. If this reading is wrong, it is the single most important thing to correct before planning proceeds.

**Interaction contract (type-toggle behavior):**
1. `#mMarketType` defaults to `"binary"` on page load — the existing binary fieldset is visible by default, pixel-identical to today's form (MARKET-03 regression guard extends to the creation form itself, not just the ticket display).
2. On change, exactly one of the three fieldsets is visible; the other two are hidden (not removed from DOM — hidden fields must not be validated/submitted).
3. Selecting "Múltipla escolha" seeds `.options-list` with exactly 2 empty rows (the enforced floor — RESEARCH.md Assumption A2).
4. "+ Adicionar opção" appends a row; disabled once 20 rows exist (RESEARCH.md Assumption A1, matches server-side MARKET-05 bound) — helper text below the list reads "Mínimo 2, máximo 20 opções." (muted, 13px, matching the existing scheduling-hint paragraph style at `.panel` line ~51 of `admin.html`).
5. Remove-row button is disabled (not hidden) when exactly 2 rows remain, preventing the floor from being violated client-side.
6. `#mThreshold` gets a non-blocking helper hint (not a validation error) suggesting `.5` increments to avoid tie outcomes — [UI-SPEC discretion, per RESEARCH.md Assumption A3]: "Dica: use .5 (ex: 2.5) pra evitar empate exato." Whole numbers are still accepted; this is copy guidance, not a client or server block.
7. Submit ("Criar mercado") always fires the same handler; the payload shape branches on `#mMarketType`'s value (`odds_yes/odds_no` vs `threshold + odds_over/odds_under` vs `options: [{label, odds}, ...]`) — server is the sole source of truth for all validation (MARKET-04), client-side checks above are UX convenience only and must never be the only gate.

## Admin Panel — Market List / Resolve Contract

The existing table (`Pergunta | Odds | Status | Agenda | Resultado | Ações`) gains one new column and generalizes two existing ones:

| Column | Binary (unchanged) | Over/Under | Multiple-choice |
|--------|--------------------|-----------|--------------------|
| **Tipo** *(new column, inserted after Pergunta)* | "Binário" | "Over/Under" | "Múltipla escolha" | — rendered with the existing `.tag` class (neutral border/text, no new color — reuses the un-modified base `.tag` style, not `.tag.admin`'s gold variant) |
| Odds | `m.odds_yes.toFixed(2)}x / ${m.odds_no.toFixed(2)}x` — **unchanged** | `"Over {threshold}: {oddsOver}x / Under {threshold}: {oddsUnder}x"` | Comma-joined `"{label} {odds}x"` per option, e.g. `"Time A 2.00x, Time B 3.50x, Empate 4.00x"` — same `.mono` `<td>`, wraps naturally via the existing `.table-wrap` horizontal-scroll container, no new CSS needed |
| Resultado | `Sim`/`Não`/`—` — unchanged | Winning option's label once resolved, else `—` | Same |
| Ações | **Unchanged**: "Fechar" (if open) + "Resolver: Sim"/"Resolver: Não" ghost buttons (if not resolved) + "Deletar" | "Fechar" (reused) + `.resolve-select` (populated with this market's own option labels) + single "Resolver" ghost button + "Deletar" (reused) | Same as Over/Under |

**[UI-SPEC decision]:** Non-binary markets use a **select + single resolve button**, not one ghost button per option. Rationale: binary's two-button pattern does not scale to up to 20 options in a table cell (`Ações` cell would overflow catastrophically and become unusable) — a dropdown keeps the row compact regardless of option count, while binary's existing, already-reviewed two-button UX stays completely untouched (MARKET-03).

## Dashboard — Option Rendering Contract

**Regression guard first:** for any market where `market.market_type === 'binary'`, `ticketTemplate()`'s output must be byte-identical to today's — same two `.odd-btn.sim`/`.odd-btn.nao` buttons, same labels ("Sim"/"Não"), same `.odds-row` (no `.multi` modifier applied). This is the direct implementation of MARKET-03 at the rendering layer; do not introduce a single shared code path that changes binary's markup as a side effect of generalizing the other two types.

**Generalized (non-binary) rendering:**
1. `market.options[]` (array from the N+1-safe `json_agg` query, RESEARCH.md Pattern 3) drives one `.odd-btn.option` per entry, each showing `option.label` (upper area, matches existing `.odd-btn .label` styling) and `option.odds.toFixed(2)}x` (matches existing `.odd-btn .value` styling) — neutral color per the Color contract above.
2. `.odds-row` gets the `.multi` modifier class whenever `market.options.length > 2`, enabling wrap so 3–20 options lay out as a responsive multi-row grid of buttons instead of squeezing into one flex row.
3. Selection state reuses the existing `selections[marketId]` object unmodified in shape — for non-binary markets its value is the chosen `option.id` (number) instead of `'yes'`/`'no'` (string). The existing "select then reveal amount input + Apostar button" flow (`.wager-form`, `.btn-primary`) is otherwise **unchanged** for all market types.
4. Wager submission payload branches: binary sends `{ market_id, choice }` (unchanged); Over/Under and multiple-choice send `{ market_id, option_id }`.
5. `loadMyWagers()` and any other row rendering `w.choice === 'yes' ? 'Sim' : 'Não'` must branch: if `w.option_id` is present, render `escapeHtml(w.option_label)` (already available via the extended `SELECT_WITH_MARKET` join, RESEARCH.md Code Examples); otherwise keep the exact existing binary ternary. Every one of the four hardcoded call sites RESEARCH.md's Pitfall 4 identifies (`admin.js` `loadMarkets()`, `dashboard.js` `ticketTemplate()`, `loadMyWagers()`, `showUserWagers()`) must apply this same branch — no call site left un-generalized.
6. No new market-type badge is added to the dashboard ticket itself — [UI-SPEC discretion] the option layout (2 buttons labeled Sim/Não vs. N labeled buttons) is self-explanatory to end users without extra chrome, and omitting it keeps the binary ticket's visual footprint completely unchanged, reinforcing MARKET-03.
